# Android 下一版本规划（对标 remodex）

## 目标

基于当前 `findeck` Android 客户端与 `remodex` 的对比，确定下一版本的产品方向、技术边界、实施顺序与验收标准。

这份规划只对标 `remodex` 的交互质量和体验闭环，不照搬其 iPhone + relay + trusted reconnect 架构。`findeck` 仍保持当前后端中心模型：

- `apps/server` 继续作为状态源
- Android 继续基于 REST + SSE
- 不在本版本引入 QR 配对、relay、端到端桥接重连

建议将下一版本定义为 `v0.2.0`，因为这次改动不是补丁级修修补补，而是一次明确的体验升级。

## 对比结论摘要

### 1. 语音按钮发送消息

结论：

- 当前 app 还不能真正使用语音按钮发送消息
- 现状只是一个禁用占位按钮
- 当前 APK 没有 `RECORD_AUDIO` 权限、没有语音识别链路、也没有录音或转写能力

对标 `remodex`：

- `remodex` 已有完整语音链路：手机录音、转写、回填到输入框、再发送
- 它做的是“语音转文字输入”，不是全双工语音对话

下一版本建议：

- 下一版本实现“语音转文字发送”MVP
- 不做实时语音聊天，不做边说边回，不做复杂语音会话态
- 推荐优先走 Android 原生语音识别 MVP，先把可用性做出来

原因：

- 接入成本最低
- 与 Android 系统语言天然一致，便于中英双语切换
- 不依赖我们先补一条新的服务端转写 API
- 能最快把“麦克风按钮可用”这件事真正交付

MVP 范围：

- 点击麦克风开始识别
- 识别完成后把文本写入 composer
- 默认由用户确认后再发送
- 可预留“识别后自动发送”开关，但不作为首发必须项

后续增强项：

- 再升级为“录音波形 + 本地录音 + 服务端统一转写”
- 这样未来 Web 端也能复用同一条语音能力

### 2. 手机版长时间锁屏后出现红色警告框

结论：

- 当前体验确实需要优化
- 锁屏在移动端是常态，不应该被 app 处理成强报错心智

当前问题本质：

- Android 端已经有 SSE 断开后的轮询补齐逻辑
- 但没有明确的前后台生命周期策略
- 长时间锁屏或后台后，连接中断容易被抬升成 `error`
- 页面层还会把部分错误展示成明显的红色错误条

对标 `remodex`：

- `remodex` 把“暂时失联”视为恢复态，而不是失败态
- 它有更清晰的 reconnect copy、恢复卡片、后台完成通知与冷启动回流

下一版本建议：

- 目标不是“锁屏后永不断线”
- 目标是“锁屏后不惊扰、解锁后可恢复、内容可补齐”

产品策略：

- 锁屏/后台时，将连接状态降级为“后台挂起/等待恢复”，而不是错误
- 回到前台后自动执行：
- 重新拉取 live run
- 重新建立 SSE
- 补齐锁屏期间消息与运行状态
- 若任务已结束，展示“已补齐后台期间结果”的中性提示

UI 策略：

- 红色错误只保留给真正需要用户处理的场景：
- 登录失效
- 会话不存在
- 主机不可达且多次重试失败
- 锁屏导致的连接丢失，改成中性或 warning 级恢复卡片

本版本不承诺：

- 长时间锁屏下持续实时流不断开
- 类似 IM 的系统级后台常驻连接

本版本承诺：

- 锁屏后不再频繁给用户“出错了”的感受
- 解锁后能自动恢复和补齐

### 3. 开屏动画完全汉化

结论：

- 可以立即做
- 这是低风险、高感知收益项

当前问题：

- `SplashScreen` 同时混用了中文与英文文案
- 这会让整体完成度打折

下一版本建议：

- 启动页文案全部资源化
- 中文系统显示全中文
- 英文系统显示全英文
- 动效逻辑不变，先把语言一致性补齐

### 4. APK 中英双语，按系统语言自动切换

结论：

- 完全可做
- 但不是只改启动页，而是一次项目级国际化改造

当前问题：

- 目前 `strings.xml` 基本只有 `app_name`
- 大量文案直接写在 Kotlin 文件里
- 还有若干日期、时长、状态文案直接绑定了中文格式

这意味着：

- 现在并不具备真正的系统语言自动切换能力
- 需要先把硬编码文案迁移到资源层

下一版本建议：

- 把双语作为本版本基础工程来做
- 范围覆盖：
- 启动页
- 登录
- 服务器列表
- 新建会话
- 会话详情
- 收件箱
- 设置
- 错误提示、状态提示、时间文案、动作文案

实施原则：

- `values/` 放默认语言
- 新增 `values-zh-rCN/`
- 或者保留中文默认，再补 `values-en/`
- 统一使用 `stringResource(...)`
- 时间、日期、数量、时长改为 `Locale.getDefault()` 驱动

### 5. 会话详情页加载时做一点有趣的小动效

结论：

- 可以做，而且主要是 Android 客户端工作
- 当前已经有 shimmer 基础，但还不够“有记忆点”

当前基础：

- 页面已有 `ShimmerBlock`
- 回复区已有少量流式光标和内容动画
- 但整页加载态仍偏“功能正确”，不够“体验成立”

下一版本建议：

- 增加轻量、克制、信息型的小动效，不追求花哨

推荐方案：

- 首屏加载卡片拆成 3 段状态：
- 正在连接主机
- 正在恢复实时流
- 正在整理会话内容
- 骨架屏从单纯灰块改成“时间线式骨架”
- 最近一条用户消息位置先出现，再补 assistant 区域
- 如果是从后台恢复，顶部控制条做一次短暂 pulse，提示“已恢复同步”

设计原则：

- 动效要帮助理解“系统正在做什么”
- 不是为了炫技
- 总时长短，不卡住用户进入内容

### 6. 参照 remodex 还可以借鉴什么

适合借鉴、且与当前架构兼容的点：

- 连接恢复卡片：把断连从“报错”改成“恢复流程”
- 后台结果提醒：任务完成、需要注意、需要补操作时给出轻提醒
- 语音录制胶囊：后续如果升级语音能力，可加入波形/时长/取消入口
- 附件入口优化：除了文件选择器，增加拍照/相册直传
- Git 能力补齐：在现有 `createBranch` / `checkoutBranch` / `commit` / `push` 基础上，后续可考虑 `pull` / `stash` / 最近提交记录
- 更细的恢复文案：例如“正在重连”“正在补齐中断期间内容”“本轮已结束，结果已同步”

不建议直接照搬的点：

- relay / trusted reconnect / QR 配对
- iOS 特有的 APNs 推送流程
- 围绕桥接安全会话设计的整套 transport 能力

## 版本范围定义

## P0：本版本必须完成

- 语音按钮 MVP 可用
- 锁屏/后台恢复体验优化
- 启动页完全双语化
- APK 基础双语化
- 会话详情页加载态升级

## P1：本版本建议完成

- 会话恢复提示卡片统一化
- 前后台切换后的自动补齐提示
- 关键页面英文文案补齐

## P2：本版本不强求，但建议进入后续 backlog

- 后台完成通知
- 拍照/相册附件
- 更完整 Git 工具栏
- 录音波形胶囊与统一转写能力

## 实施分期

### Phase 1：国际化基础层

目标：

- 为双语和启动页改造打底

Android 改动：

- 新增 `res/values-en/strings.xml`
- 新增 `res/values-zh-rCN/strings.xml`
- 把高频页面文案从 Kotlin 硬编码迁到资源
- 清理固定中文日期/时长格式

重点文件：

- `apps/android/app/src/main/res/values/strings.xml`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/...`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/sessions/SessionDetailHelpers.kt`

验收标准：

- 系统切英文时，主流程页面不再出现中文硬编码
- 系统切中文时，主流程页面不再出现英文残留
- 启动页语言与系统保持一致

### Phase 2：锁屏/后台恢复体验

目标：

- 把锁屏断连从“错误体验”改成“可恢复体验”

Android 改动：

- 引入 app 前后台生命周期感知
- 后台时收敛错误上抛
- 前台恢复时自动刷新 live run + session detail + SSE
- 错误条与恢复卡片分层

重点文件：

- `apps/android/app/src/main/java/app/findeck/mobile/MainActivity.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/sessions/SessionDetailScreen.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/sessions/SessionDetailViewModel.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/data/network/ApiClient.kt`

验收标准：

- 锁屏 15 分钟后恢复，不出现误导性的红色报错
- 若任务仍在运行，页面可继续补齐
- 若任务已结束，页面可自动落到最终状态
- 只有真正不可恢复的错误才显示 error 风格

### Phase 3：语音按钮 MVP

目标：

- 让麦克风按钮真正可用

推荐技术路线：

- Android 原生语音识别优先
- 将识别结果写入 composer
- 首发不做服务端统一转写

Android 改动：

- 增加录音权限
- 新增语音输入控制器
- 连接 composer 与发送链路
- 增加识别失败/权限关闭的恢复提示

重点文件：

- `apps/android/app/src/main/AndroidManifest.xml`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/sessions/ComposerBar.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/sessions/SessionDetailScreen.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/sessions/SessionDetailViewModel.kt`

验收标准：

- 麦克风按钮不再是灰色占位
- 首次使用时可正确请求权限
- 识别成功后文本进入输入框
- 用户可继续编辑后发送
- 网络差、权限拒绝、设备不支持时有清晰兜底提示

### Phase 4：启动页与会话详情动效升级

目标：

- 提升第一屏与详情页的完成度

Android 改动：

- 启动页文案资源化
- 启动页语言按系统自动切换
- 会话详情增加分阶段加载态
- 后台恢复增加轻提示动效

重点文件：

- `apps/android/app/src/main/java/app/findeck/mobile/ui/splash/SplashScreen.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/sessions/SessionDetailScreen.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/sessions/MessageBubbles.kt`

验收标准：

- 启动页中英文完全分离
- 会话详情加载态更有引导性
- 动效不影响输入和滚动性能

## 推荐执行顺序

1. 先做国际化基础层
2. 再做锁屏/后台恢复
3. 然后做语音按钮 MVP
4. 最后做启动页与加载动效润色

原因：

- 国际化会改很多文案，如果后做，前面新增能力还要返工一次
- 锁屏恢复是当前最直接的体验痛点
- 语音按钮是高感知功能，但依赖文案与错误恢复
- 动效适合放在功能稳定后统一收尾

## 风险与边界

### 风险 1：Android 后台限制

说明：

- Android 不适合承诺“锁屏后长期持续实时连接”
- 我们应该承诺“恢复快、补齐准、提示轻”

应对：

- 以恢复和 catch-up 为主
- 不把后台断流本身当成失败

### 风险 2：双语改造范围大

说明：

- 当前 Kotlin 里硬编码文案较多
- 如果只补启动页，会形成局部双语、整体单语的割裂

应对：

- 本次按页面分批清理
- 主链路页面必须全部资源化

### 风险 3：语音识别设备差异

说明：

- Android 原生语音识别在不同 ROM、Google 服务可用性、网络状态下表现不一致

应对：

- 首发把语音识别定义为 MVP
- 失败时允许回退到手动输入
- 后续再评估服务端统一转写

## 测试矩阵

- 中文系统 + 英文系统
- 首次安装 + 已登录用户升级
- 锁屏 5 分钟 / 15 分钟 / 30 分钟
- 锁屏前会话空闲 / 正在运行 / 已结束未同步
- 语音权限允许 / 拒绝 / 永久拒绝
- Wi-Fi 切换 / 网络中断 / 恢复
- 长会话详情页加载与滚动性能

## 最终版本验收标准

- 用户第一次打开 app，启动页语言与系统一致
- 用户进入会话详情时，加载态更清晰、更有质感
- 用户长时间锁屏后再打开，不会被红色错误打脸
- 用户可以通过麦克风把语音转成消息文本并顺利发送
- 中文和英文系统下，主流程页面都达到可发布水平

## 建议的版本结论

下一版本建议聚焦一句话：

“把 Android 客户端从能用的远程控制台，提升到真正适合日常锁屏使用、支持双语、并具备语音输入能力的移动端工作台。”

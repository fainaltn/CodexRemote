# 会话切换体验验收清单

这份清单用于验收本轮 `Phase 4：长历史渲染优化` 与 `Phase 5：导航和状态设计收口` 的实际落地效果。

对应计划书：

- [docs/paseo-gap-improvement-plan.md](/Users/fainal/Documents/GitHub/CodexRemote/docs/paseo-gap-improvement-plan.md)

这份清单只覆盖“会话切换体验”主线，不替代语音、文件、审批、repo 操作等其他功能线的专项验收。

## 一、验收口径

本轮通过标准不是“某个接口能返回”，而是下面两件事是否同时成立：

1. 长历史会话已经从“整块重载”转成“尾部优先、逐步展开”
2. 列表到详情、前台到后台恢复、弱网到恢复同步，这些状态已经具备更明确的连续感和状态表达

## 二、建议测试样本

建议至少准备下面这些样本：

- Web：桌面浏览器 1 个
- Android：真机 1 台
- 短历史会话 1 个
- 长历史会话 1 个
- 正在运行中的会话 1 个
- 弱网或短暂断网场景 1 组

如果要给最终说明背书，建议把“长历史会话”再分成：

- 中等长度会话
- 明显超长会话

## 三、Phase 4：长历史渲染优化验收

### 1. Web 长历史首屏

1. 打开一个明显较长的已有会话。
通过标准：
- 页面优先出现会话壳层、标题、路径与最近历史
- 默认不是等待整段完整历史返回后才显示主内容
- 历史区能明确显示 `部分历史` 或 `完整历史`

2. 在同一个会话里观察当前轮和历史轮。
通过标准：
- 当前轮内容优先可读
- 历史内容不是一整块完全平铺
- 历史分组或折叠态不会让当前轮被误认为历史内容

3. 点击“加载更早回合”。
通过标准：
- 更早历史继续向前展开
- 已有内容不丢失、不重排错乱
- 页面不会退回整页 loading

### 2. Android 长历史首屏

1. 打开一个明显较长的已有会话。
通过标准：
- 顶栏、控制条、最近历史优先出现
- 历史区能显示 `部分历史` 或 `完整历史`
- 默认不是把整段历史一次性塞满后才可用

2. 观察历史轮和当前轮的层次。
通过标准：
- 历史轮和当前轮区分清楚
- 当前轮内容优先可读
- 折叠内容的展开不会打断当前阅读位置

3. 点击“加载更早消息”。
通过标准：
- 会继续向前分页展开
- 当前上下文不丢失
- 页面不掉回整页加载态

### 3. 活跃会话下的长历史收束

1. 打开一个正在运行、同时历史较长的会话。
通过标准：
- 当前轮仍然优先可见
- 运行中不会频繁因为历史补齐而整页重建
- 终态收束优先通过轻量 tail 补齐，只有必要时才退回更重的刷新

### 4. Phase 4 不通过条件

出现以下任一情况，建议判定 Phase 4 仍未通过：

- 长历史会话进入时仍然明显卡死
- 点击进入后要长时间等待整块历史返回
- “加载更早”会导致已有历史丢失、跳位或顺序错乱
- 当前轮在长历史场景下不够稳定，容易被历史内容淹没

## 四、Phase 5：导航和状态设计收口验收

### 1. Web 列表到详情切换

1. 从侧栏或 `sessions` 页进入已有会话。
通过标准：
- 进入前后能感知到目标会话正在被切入
- 目标会话的壳层、标题、路径能优先出现
- 不再是完全冷启动的整页等待

2. 连续切换两个不同会话。
通过标准：
- 切换方向清晰
- 上一会话内容不会长时间误导成当前会话
- 过渡提示是短暂的，不会滞留

3. 返回列表观察状态层级。
通过标准：
- `当前会话`、`当前项目`、`最近活跃` 的识别是清楚的
- 状态强调不只靠很弱的颜色差异

### 2. Android 列表到详情切换

1. 从会话列表进入已有会话。
通过标准：
- 列表侧的项目层级和最近活跃信息清楚
- 进入详情时不是纯粹的整页 loading card
- 命中预热缓存时，壳层与最近上下文会优先出现

2. 返回列表，再切到另一个会话。
通过标准：
- 切换感应当接近“进入同一工作区里的另一条线程”
- `当前会话` 与 `当前项目` 层级保持稳定
- 不出现明显白屏或长时间空等

3. 观察详情页过渡反馈。
通过标准：
- 切换中提示是存在且短暂的
- 过渡完成后不会残留错误提示或无意义 banner

### 3. 恢复同步 / 降级同步状态

1. Web：在运行中的会话里制造一次短暂断网，再恢复。
通过标准：
- 恢复中或降级中的状态对用户是可见的
- 当前内容不丢
- 恢复后会继续补齐，而不是整页重开

2. Android：在运行中的会话里切到后台再返回，或模拟短暂断网。
通过标准：
- 可以观察到 `恢复同步`、`降级同步`、`已同步` 这类状态变化
- 已有内容不丢失
- 恢复后以补齐为主，而不是整页重置

### 4. 状态语言一致性

需要人工确认下面这些词在 Web / Android 上含义一致：

- 当前会话
- 当前项目
- 最近活跃
- 部分历史
- 完整历史
- 恢复同步
- 降级同步
- 已同步

通过标准：

- 同一状态在不同端不应出现明显冲突的叫法
- 同一层级状态的强调方式应大致一致
- 恢复/降级应表现为“过程状态”，不是“错误替代品”

### 5. Phase 5 不通过条件

出现以下任一情况，建议判定 Phase 5 仍未通过：

- 切换状态仍然主要靠整页 spinner 或纯错误提示表达
- 切换中提示滞留不消失
- 当前会话 / 当前项目 / 最近活跃层级不清楚
- 弱网或后台恢复后仍然容易整页重开

## 五、诊断辅助项

这部分不是强制通过项，但建议在人工验收时顺手记录：

1. Web / Android 是否都能看到“先壳层、后补齐”的明显差异
2. Android 诊断面板里的 `Bootstrap load / Summary load / Tail load` 是否能稳定出数
3. 长历史与短历史相比，等待时间是否已经明显缩短

注意：这些观察可以支撑“体验明显改善”，但不等于正式 benchmark。

## 六、不应写进最终说明的夸大表述

如果只完成本清单而没有额外专项验证，最终说明里仍然不建议写：

- “长历史性能问题已彻底解决”
- “深层虚拟化已完成”
- “跨端状态组件体系已全部统一”

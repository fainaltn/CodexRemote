# v0.3.0 Roadmap

## Goal

`v0.3.0` is the release where CodexRemote should start feeling like a deliberate product instead of a capable collection of tools.

This roadmap narrows the next version to the five directions that were selected after comparing CodexRemote with `remodex` on April 15, 2026:

- one-click startup and operator entry
- stronger first impression and product feel
- background recovery and weak-network state handling
- notification tiers instead of a single completion alert
- more explicit advanced controls on the session surface

This is an execution document. It does not reopen the transport-architecture discussion.

## Product Position

CodexRemote `v0.3.0` should feel like a calm remote Codex cockpit:

- fast to start
- safe to leave in the background
- clear when live state degrades
- polished on first launch and first login
- strong enough for real session steering from Android and web

## Non-Goals

The following remain out of scope for `v0.3.0`:

- QR pairing
- relay or trusted reconnect architecture
- end-to-end encrypted phone-to-host transport work
- iOS parity or an iPhone-first product rewrite
- major backend redesign that is not required by the chosen UX work

## Release Shape

The recommended delivery order is:

1. one-click startup and operator entry
2. background recovery and weak-network UX
3. first impression and product feel
4. notification tiers
5. advanced-control promotion

This order keeps the release grounded in reliability and product closure before expanding the control surface.

## P0

### 1. Unified Startup And Operator Entry

#### Why

Today CodexRemote is operable, but the entry story is still split across `scripts/`, workspace commands, and launchd instructions. `remodex` has a clearer operational story. We should not copy its transport model, but we should match its clarity.

#### Outcome

Ship a single repo-local entrypoint that covers the common operator flows:

- `codexremote up`
- `codexremote status`
- `codexremote logs`
- `codexremote restart`
- `codexremote web`
- `codexremote doctor`

For `v0.3.0`, this can ship as a repo-local wrapper before we decide whether to promote it into a packaged CLI.

#### File Scope

- `package.json`
- `scripts/codexremotectl.sh`
- `scripts/codexremote-server.sh`
- `scripts/codexremote-web.sh`
- new `scripts/codexremote.sh` wrapper
- `README.md`
- `docs/operations.md`
- `scripts/README.md`

#### Acceptance Criteria

- A developer can start the local stack with one command from the repo root.
- Service status and logs are discoverable without reading multiple docs first.
- `doctor` can catch the most common local setup mistakes, including missing `.env.local`, missing `CODEXREMOTE_PASSWORD`, and missing built workspaces or launchd files.
- The README and operations docs tell the same startup story.

### 2. Background Recovery And Weak-Network State Model

#### Why

The goal is not to keep a mobile real-time stream alive forever. The goal is to make interruption feel recoverable instead of broken.

CodexRemote already has the technical building blocks:

- Android-side polling fallback
- SSE reconnect logic on web
- session refresh flows
- background completion notification

What is still missing is a deliberate state model and calmer UI language.

#### Outcome

Reframe transport loss as a recovery flow:

- background or lock-screen disconnects should not immediately surface as hard errors
- foreground resume should automatically refresh session detail and live run state
- the UI should distinguish `recovering`, `degraded`, and `failed`
- if the run finished while the app was backgrounded, show a neutral catch-up message instead of a red warning

#### File Scope

- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/SessionDetailViewModel.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/SessionDetailScreen.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/ConversationTimeline.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/TimelineStatusCards.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/MicroStatusRow.kt`
- `apps/android/app/src/main/res/values/strings_session_detail.xml`
- `apps/android/app/src/main/res/values-en/strings_session_detail.xml`
- `apps/web/src/lib/use-sse.ts`
- `apps/web/src/app/sessions/[sessionId]/page.tsx`
- `apps/web/src/app/globals.css`
- optional backend follow-up only if new recovery metadata is truly required

#### Acceptance Criteria

- Locking the phone or backgrounding the app no longer produces a default red failure state for recoverable disconnects.
- Returning to the foreground automatically attempts refresh and stream reattachment.
- Users can tell the difference between temporarily reconnecting, recovered-and-caught-up, and genuinely failed states that need action.
- Web and Android use the same recovery vocabulary.

## P1

### 3. First Impression And Product Feel

#### Why

The current product already works, but the first-launch and first-login story does not yet establish confidence the way the session detail experience does.

The gap is especially visible in:

- login
- server list empty state
- session list loading and empty states
- session detail initial loading and restore states

#### Outcome

Turn the first-use flow into a coherent `Precision Console` experience:

- stronger hero and setup framing
- clearer host/workspace story
- better empty states and restoring states
- more deliberate loading language
- more obvious distinction between static history and live execution

#### File Scope

- `apps/web/src/app/login/page.tsx`
- `apps/web/src/app/page.tsx`
- `apps/web/src/app/app-shell-client.tsx`
- `apps/web/src/app/sessions/page.tsx`
- `apps/web/src/app/sessions/[sessionId]/page.tsx`
- `apps/web/src/app/globals.css`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/login/LoginScreen.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/servers/ServerListScreen.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/splash/SplashScreen.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/SessionListScreen.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/SessionDetailScreen.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/theme/*`

#### Acceptance Criteria

- The login and empty-state flows sell a clear product story instead of feeling like utility pages.
- Initial session loading feels intentional and informative.
- The first-use surfaces across web and Android share the same hierarchy and product language.
- The visual direction stays inside `Precision Console` and does not drift into a different brand system.

### 4. Notification Tiers

#### Why

CodexRemote already posts a background run completion notification on Android. The next step is to stop treating all background events as the same kind of attention.

#### Outcome

Introduce notification tiers with clear user meaning:

- run completed
- run failed
- needs attention
- recovered and synced

The notification system should deep-link back into the right session and avoid noisy duplicate alerts.

#### File Scope

- `apps/android/app/src/main/java/dev/codexremote/android/notifications/RunCompletedNotificationHelper.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/notifications/RunCompletedNotificationModels.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/MainActivity.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/SessionDetailViewModel.kt`
- `apps/android/app/src/main/res/values/strings_notification.xml`
- `apps/android/app/src/main/res/values-en/strings_notification.xml`
- optional server-side metadata only if the client cannot reliably derive attention level from existing run state

#### Acceptance Criteria

- Background notifications map to distinct outcomes instead of a single generic completion path.
- Tapping a notification routes back into the relevant session detail.
- A completed run, a failed run, and a recovery-sync event do not share identical copy.
- Notification logic remains quiet when the user is already active in the foreground session.

## P2

### 5. Advanced Controls As A Visible Control Surface

#### Why

CodexRemote already exposes runtime controls for model and reasoning, plus queueing and repo actions. The next gap versus `remodex` is not pure feature count but control visibility and user confidence.

We should first promote existing capability before deciding how much new capability to expose.

#### Outcome

Phase the work in two steps.

Step 1 for `v0.3.0`:

- make current runtime controls more legible
- show active runtime configuration more clearly
- clarify how queued prompts inherit or override current controls

Step 2 for capability-gated evaluation:

- plan-mode entry
- access-mode entry
- subagent-oriented entry points

Any new control added here must be backed by real backend support and clear failure semantics. We should not ship decorative controls that only imply capability.

#### File Scope

- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/ComposerRuntimeControls.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/ComposerBar.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/SessionControlStrip.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/ui/sessions/SessionDetailViewModel.kt`
- `apps/android/app/src/main/java/dev/codexremote/android/data/network/ApiClient.kt`
- `packages/shared/src/api/live-runs.ts`
- `packages/shared/src/schemas/run.ts`
- `apps/server/src/routes/live-runs.ts`
- `apps/web/src/app/sessions/[sessionId]/page.tsx`
- `apps/web/src/lib/api.ts`

#### Acceptance Criteria

- A user can see the active model and reasoning state at a glance.
- Runtime choices feel like part of the control surface, not hidden configuration.
- Queued prompts make their control inheritance or override state visible.
- Any newly added advanced control is capability-gated and clearly explained when unavailable.

## Cross-Cutting Requirements

### Shared State Vocabulary

Web and Android should converge on the same state words for:

- connecting
- streaming
- queued
- recovering
- recovered
- degraded
- failed

### No Hidden Architecture Drift

If a roadmap task appears to require backend changes, the default answer is to minimize them.

Only introduce backend or shared-schema work when one of these is true:

- the UI cannot represent the state accurately without new data
- the feature would otherwise require duplicated inference in every client
- the operator story would remain incomplete without it

### Documentation Closure

By the end of `v0.3.0`, the repository should have one coherent story for:

- how to start the stack
- how to inspect service state
- what happens when live streaming degrades
- what the Android notifications mean
- which advanced controls are intentionally exposed

## Suggested Delivery Slices

Ship in this order:

1. repo-local unified CLI wrapper and operations doc cleanup
2. Android recovery-state refactor
3. web recovery-state parity
4. login and session-loading polish
5. notification tiers
6. runtime-control promotion

This keeps the release front-loaded with clarity and reliability, while still leaving room for product polish and control-surface upgrades.

# v0.5.x Next Phase Plan

## Status

Proposed on 2026-04-19.

Update on 2026-04-19:

- `v0.5.1` is ready to close.
- `v0.5.2` is ready to close.
- `v0.5.3` is now the active implementation target.

This document is the execution plan for the next phase after `v0.5.0`.

It does not reopen the transport architecture discussion. The goal is to turn the freshly unified `findeck` baseline into a more reliable daily tool:

- pairing and trusted reconnect should feel predictable
- operator diagnostics should point to the real failure source fast
- Web should become a desktop-first fallback workbench
- release and APK delivery should cost less operator attention
- legacy naming cleanup should happen as a migration task, not as random incidental edits

## Baseline

- Workspace version: `0.5.0`
- Android `versionName`: `0.5.0`
- Brand unification is complete enough to use as the new baseline
- Android APK is already the primary mobile control surface
- Web no longer needs to optimize for phone-shell constraints first

## Product Position For This Phase

### Surface roles

- `apps/android`: the primary mobile remote-control client
- `apps/web`: a desktop-first browser workbench for login, session review, live follow-up, inbox handling, and repo-adjacent control
- `apps/server`: the source of truth for auth, trusted pairing, session state, inbox state, and live run state

### Web direction

For this phase, Web is not treated as a mobile-first shell.

The Web direction is:

- desktop-first layout
- clearer project and session hierarchy
- less hero/marketing framing
- stronger workbench behavior
- calmer but denser operator-visible state

### Boundary Reminder

This phase is explicitly about tightening the product, not expanding it without limit.

The guiding rule is:

- improve the clarity, reliability, and usability of the existing control surface
- do not keep adding low-frequency capabilities just because Web can hold more UI

For Web specifically:

- align key control-surface experience with Android where that reduces user confusion
- do not turn Web into a heavier parallel product with its own feature agenda
- do not rebuild a full desktop IDE, full Git client, or a second orchestration system inside the browser

The target shape remains a lightweight remote-control workbench, not a maximal feature dashboard.

## External Reference Conclusions

We only borrow interaction and product-shape lessons from the reference products. We do not copy their full architecture.

| Reference | Borrow | Do Not Borrow | Meaning For findeck |
| --- | --- | --- | --- |
| `Farfield` | direct browser control surface, project-grouped thread tree, low-friction “read / send / monitor / interrupt” loop | ultra-thin feature ceiling | Web should feel more like a desktop control panel and less like a branded splash shell |
| `HAPI` | local-to-remote continuity, clear handoff model, explicit recovery language | PWA/Telegram/voice expansion as a first priority | Web should explain recovery and remote continuation better, without becoming mobile-first again |
| `Paseo` | stronger object model across session / provider / control surface, clearer workbench hierarchy | broader orchestration scope and heavier product surface | Web should become more legible and structured, but stay lightweight |

## Non-Goals

The following remain out of scope for `v0.5.x`:

- reopening relay, tunnel, or public-remote architecture
- making Web the primary mobile experience again
- turning Web into a bigger and heavier product than Android
- expanding Android into a heavier mobile IDE
- iOS parity work
- large backend redesign not required by the chosen reliability or Web work
- deleting legacy compatibility paths without a migration gate
- adding low-frequency controls that make the surface denser without improving the main workflow

## Release Shape

The recommended release train is:

| Version | Theme | Planned Outcome |
| --- | --- | --- |
| `v0.5.1` | trusted reconnect audit | pairing, claim, reconnect, and restore states become more explicit and less guessy |
| `v0.5.2` | cold launch recovery | new install, reinstall, service restart, and reconnect failure paths become recoverable instead of ambiguous |
| `v0.5.3` | operator diagnostics | `findeck doctor / status / logs` become reliable enough to identify the real failing layer quickly |
| `v0.5.4` | Web entry reset | root, login, and shell framing shift from mobile-shell leftovers to desktop-first control surface |
| `v0.5.5` | Web sessions workbench | session list, project tree, and current-session focus become the main desktop workflow |
| `v0.5.6` | Web detail and inbox workbench | desktop session detail and inbox become stable fallback paths for real follow-up work |
| `v0.5.7` | release and delivery automation | release preparation, APK artifact production, and delivery become more repeatable |
| `v0.5.8` | legacy migration guardrails | old names and compatibility layers are inventoried and classified before any breaking cleanup |
| `v0.6.0` | legacy migration execution | execute the approved cleanup only if `v0.5.8` proves the migration path is safe |

If a higher-priority reliability item slips, later Web polish versions should slide rather than jumping ahead.

## P0

### 1. Pairing, Trusted Reconnect, And Cold Launch Recovery

#### Why

The system already supports:

- one-time pairing codes
- trusted client credentials
- reconnect with `clientId` and `clientSecret`
- Android-side trusted-host storage

What still feels weak is the recovery story after:

- a fresh install
- an app reinstall
- a host password change
- a service restart
- an expired or invalid trusted secret
- a cold launch after time away

The next step is not “add more auth modes.” The next step is to make existing trusted recovery easier to understand and recover from.

#### Outcome

Ship a calmer recovery loop where:

- trusted reconnect either succeeds fast or fails with a precise reason
- cold launch can tell the difference between `host unreachable`, `password drift`, `token expired`, and `trusted credentials invalid`
- Android no longer makes the user guess whether they should wait, retry, or re-pair
- the server-side contract stays explicit enough for future Web reuse

#### Subtasks

- `v0.5.1`: audit the full `pairing -> claim -> reconnect -> persisted trusted host` chain, tighten server/client state vocabulary, and expose clearer recovery semantics
- `v0.5.2`: finish the cold-launch restore loop, including explicit re-pair fallback and regression coverage for reinstall and service-restart scenarios

#### File Scope

- `apps/server/src/routes/pairing.ts`
- `apps/server/src/pairing/store.ts`
- `apps/server/src/routes/auth.ts`
- `apps/server/src/auth/store.ts`
- `packages/shared/src/api/auth.ts`
- `packages/shared/src/schemas/trusted-client.ts`
- `apps/android/app/src/main/java/app/findeck/mobile/data/repository/ServerRepository.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/servers/PairingScreen.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/servers/ServerListScreen.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/settings/ServerSettingsScreen.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/ui/splash/SplashScreen.kt`
- `apps/android/app/src/main/java/app/findeck/mobile/MainActivity.kt`

#### Acceptance Criteria

- A trusted host can reconnect after a normal cold launch without requiring the user to guess what happened.
- Reinstall and invalid-secret cases fail with a clear re-pair path instead of a vague auth failure.
- Service restart and token-expiry cases do not silently degrade into “broken” state when trusted reconnect is still possible.
- Android and server-side logging use the same recovery language.

### 2. Credible Operator Diagnostics

#### Why

`findeck doctor`, `status`, and `logs` already exist, but they still need to become the place where an operator can tell:

- whether the backend is up
- whether Web is pointing at the right API
- whether launchd is healthy
- whether password state drifted
- whether pairing/trusted-host state is the real problem

This needs to be solved before more surface-level polish, because the system is still a local-first operator tool.

#### Outcome

Turn the operator commands into a truthful first-stop diagnostic layer instead of a partial convenience wrapper.

#### Subtasks

- `v0.5.3`: upgrade `doctor / status / logs` so they point at the actual failing layer and stop forcing the user to inspect code or multiple docs first

#### File Scope

- `scripts/findeck.sh`
- `scripts/findeckctl.sh`
- `scripts/findeck-server.sh`
- `scripts/findeck-web.sh`
- `scripts/install-launchd.sh`
- `apps/server/src/routes/health.ts`
- `apps/server/src/config.ts`
- `README.md`
- `docs/operations.md`
- `scripts/README.md`

#### Acceptance Criteria

- `doctor` can distinguish between missing env, missing build output, launchd state failure, and Web/API target mismatch.
- `status` can show whether the backend and Web are actually reachable, not just whether a helper script ran.
- `logs` gives a fast path to the relevant server/Web evidence without making the operator search manually.
- The README and operations docs still tell the same operator story as the scripts.

## P1

### 3. Desktop-First Web Control Surface

#### Why

Web has become the main leftover rough edge after the Android stabilization and brand-unification work.

The current Web surface still carries too much of the earlier “must also behave like a phone shell” pressure:

- too much explanatory chrome on the way into work
- too many overview and metric blocks competing with the main workflow
- session and project hierarchy not yet forceful enough for desktop use
- session detail behaving partly like a dashboard and not enough like a workbench

Since Android is now the complete mobile client, Web should be explicitly re-centered on desktop use.

#### Outcome

Turn Web into a desktop-first fallback workbench where a user can reliably:

- log in quickly
- find the right project and session fast
- continue a running session
- understand live vs degraded vs historical state
- handle inbox work and file intake from a desktop browser

This is not a full frontend rewrite. It is a hierarchy and workflow cleanup.

Where Web and Android overlap, the goal is consistency in the main task loop, not feature competition.

#### Subtasks

- `v0.5.4`: reset root, login, and app shell framing so Web feels like a desktop control surface instead of a mobile-shell carryover
- `v0.5.5`: restructure the session index around project tree, current session, and workbench-first layout
- `v0.5.6`: restructure session detail and inbox for desktop follow-up work, with clearer run-state handling and less dashboard noise

#### File Scope

- `apps/web/src/app/page.tsx`
- `apps/web/src/app/login/page.tsx`
- `apps/web/src/app/app-shell-client.tsx`
- `apps/web/src/app/sessions/page.tsx`
- `apps/web/src/app/sessions/[sessionId]/page.tsx`
- `apps/web/src/app/sessions/new/page.tsx`
- `apps/web/src/app/inbox/page.tsx`
- `apps/web/src/app/globals.css`
- `apps/web/src/lib/api.ts`
- `apps/web/src/lib/auth.tsx`
- `apps/web/src/lib/use-sse.ts`

#### Acceptance Criteria

- Login becomes a fast desktop entry instead of a mini product pitch.
- The session index makes project grouping and current work obvious at first glance.
- Session detail prioritizes `continue work`, `see live state`, and `recover from degradation`.
- Inbox feels usable from a desktop browser for real intake work, not just as a mobile-friendly accessory.
- Web no longer feels visually or structurally constrained by old mobile-shell assumptions.

### 4. Release, Deploy, And APK Delivery Automation

#### Why

The repository already has:

- release notes and checklist documents
- a repo-specific release skill
- an APK publish script
- a split between GitHub release work and SMB APK delivery

The next step is to reduce the manual joins between them.

#### Outcome

Make release work more repeatable so a normal release no longer requires too much manual orchestration between docs, scripts, APK naming, and GitHub handoff.

#### Subtasks

- `v0.5.7`: align release preparation, APK artifact generation, and delivery entrypoints into a cleaner automation path with explicit dry-run support

#### File Scope

- `scripts/publish-android-apk.sh`
- `skills/findeck-release/SKILL.md`
- `skills/findeck-release/scripts/release_findeck.sh`
- `skills/apk-smb-sync/SKILL.md`
- `docs/release-checklist.md`
- `docs/release-notes-v0.5.0.md`
- `README.md`

#### Acceptance Criteria

- The release path makes it obvious which step is GitHub release work and which step is APK delivery work.
- A dry run can validate the release path without producing accidental external side effects.
- APK artifact naming stays stable and predictable.
- Release docs, release skill, and publish scripts tell the same story.

## P2

### 5. Legacy Cleanup As A Gated Migration

#### Why

The repo still contains deliberate compatibility paths such as:

- `CODEXREMOTE_*` environment variable fallback
- legacy database-name fallback
- older launchd naming
- transition-period compatibility for older pairing payload shapes

These should not be removed casually during unrelated work.

#### Outcome

Treat legacy cleanup as a migration task with explicit gates:

- first inventory and classify
- then add migration warnings and operator checks
- only then execute the approved rename/removal set

#### Subtasks

- `v0.5.8`: complete the inventory, classify each legacy path as `keep`, `rename`, `migrate`, or `remove-later`, and ship guardrails
- `v0.6.0`: execute the approved cleanup only if the migration path is verified and documented

#### File Scope

- `apps/server/src/config.ts`
- `apps/server/src/auth/store.ts`
- `apps/server/src/db.ts`
- `apps/server/src/routes/auth.ts`
- `scripts/findeck.sh`
- `scripts/findeck-server.sh`
- `scripts/findeck-web.sh`
- `scripts/install-launchd.sh`
- `infra/launchd/dev.findeck.server.plist`
- `README.md`
- `docs/operations.md`

#### Acceptance Criteria

- No compatibility path is removed just because nearby code was touched.
- The repo has a written classification for the remaining legacy names.
- Operator-facing warnings exist before any breaking cleanup lands.
- The actual migration release only happens once it is demonstrably safer than keeping the compatibility layer.

## Recommended Delivery Order

1. trusted reconnect and cold-launch recovery
2. operator diagnostics
3. desktop-first Web entry and session workbench cleanup
4. release and APK automation
5. legacy migration guardrails
6. legacy migration execution only if guardrails prove it is safe

This keeps the phase grounded in reliability first, then desktop usability, then operator convenience, and only then cleanup.
